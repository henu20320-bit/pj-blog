# 爬虫之文章抓取

这次我给博客补内容时，第一次比较完整地把“抓取网页内容再整理成文章”这件事走通了一遍。仓库里本来就有一个现成的爬虫脚本，所以我没有另外搭新的系统，而是直接沿着现有工具去跑。

## 这次用到的工具

项目里已经有一个简单的抓取脚本：

- `scraper/crawl.py`
- 依赖：`crawl4ai`、`playwright`

它做的事情很直接：

1. 接收一个 URL
2. 抓取页面正文
3. 把内容转成 Markdown
4. 保存到 `scraped_data/` 目录

这套思路很适合做博客素材沉淀，因为抓取后的结果天然就是文本，不需要再从 HTML 里一点点抠内容。

## 这次实际抓了什么

这次我主要抓了两类内容：

- 学习型资料：比如 `https://xiaolincoding.com/os/`
- 博客型文章：比如 `https://paiad-blog.vercel.app/blog/codex-cli/`、`https://paiad-blog.vercel.app/blog/ai-agent/`、`https://paiad-blog.vercel.app/blog/github-homepage-diy/`

抓取完成后，原始文件都会进 `scraped_data/`，这相当于一个原料仓库。之后再决定哪些内容要进入博客、哪些内容更适合放进笔记。

## 这次踩到的坑

### 1. Windows 控制台编码问题

第一次跑爬虫时，抓取流程其实已经开始了，但日志输出被 Windows 终端的 `gbk` 编码卡住，报了字符编码错误。这个问题和网页本身没关系，主要是命令行在打印特殊字符时出错。

这次我的处理方式是给脚本执行时补一个环境变量：

```powershell
$env:PYTHONIOENCODING='utf-8'
```

这样 Python 的标准输出按 UTF-8 走，爬虫就能正常跑完。

### 2. 抓到的不一定是“可直接发布”的内容

抓取成功不等于文章就能直接发。

很多页面抓下来后会夹带这些内容：

- 顶部导航
- 搜索框说明
- 页脚
- 评论区
- 目录锚点
- 外链按钮

所以抓取后的 Markdown 更像“素材稿”，不一定适合直接当博客正文。我这次的做法是：

- 原始抓取文件留在 `scraped_data/`
- 正式发布到博客的内容再做一轮人工整理

### 3. 资料和博客要分开管理

这次还让我意识到一个很实际的问题：

- 有些内容更适合放“博客”
- 有些内容其实是“笔记”或者“资料整理”

所以我后来把站点拆成了两个板块：

- `blog/`：更偏成品化、可直接阅读的文章
- `notes/`：更偏学习沉淀、资料整理、抓取笔记

这样后面继续扩展内容时，结构会更清楚。

## 我现在比较认可的流程

如果后面继续批量给博客补内容，我会优先按这套流程来：

1. 先抓取目标网页
2. 把原始 Markdown 存进 `scraped_data/`
3. 判断它更适合放博客还是笔记
4. 再整理成带 frontmatter 的正式文章
5. 最后通过本地构建确认页面能正常生成

## 为什么这个流程值得保留

它最大的优点不是“快”，而是“可复用”。

只要目标网页能稳定抓下来，后面无论是：

- 继续补博客内容
- 做专题资料整理
- 沉淀成笔记库
- 批量建立学习索引

都可以复用同一套基础流程。

## 小结

这次让我真正把“爬虫抓资料”和“博客内容生产”接上了。爬虫负责把网页变成可编辑的文本，博客负责把文本变成适合长期阅读的内容。前者解决采集，后者解决表达，这两步分开之后，整件事就顺了很多。
