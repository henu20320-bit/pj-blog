# Codex CLI 基础使用

Codex CLI 是一个面向终端工作流的 AI 编码助手。它不是单纯的聊天窗口，而是直接运行在本地命令行里的协作工具，可以围绕项目目录做读取、修改、执行和检查。

## 适合什么场景

如果你平时的工作方式本来就是围绕终端展开的，那 Codex CLI 会很顺手。它比较适合这些场景：

- 在已有项目里快速理解结构
- 直接让 AI 帮你改文件
- 跑构建、测试、检查日志
- 一边操作代码，一边让 AI 解释问题
- 对一组改动做快速 review

它更像一个“会配合你干活的终端搭子”，而不是只负责回答问题的模型窗口。

## 基本要求

目前常见的平台环境包括：

- macOS
- Linux
- Windows

如果是在 Windows 上使用，终端环境和编码配置最好保持稳定，必要时可以优先考虑 WSL，这样命令行体验会更顺。

## 进入项目后的常见命令

启动后，最常见的是这几类交互命令：

```text
/init
/status
/approvals
/model
/review
```

### `/init`

初始化项目说明文件，帮助 AI 明确当前仓库的协作规则。

### `/status`

查看当前会话状态，比如模型、权限、目录上下文之类的信息。

### `/approvals`

管理命令执行权限。这个选项很重要，因为它决定了 AI 在当前会话里能自动做到什么程度。

### `/model`

切换模型或调整推理强度。不同任务对模型速度和推理深度的需求不一样，这个命令能帮助你找到更合适的平衡点。

### `/review`

更适合拿来做代码审查，尤其是在你已经完成一批改动之后，让它专门从风险和回归角度再看一遍。

## 一种比较实用的使用流程

我比较推荐的工作流是这样的：

1. 先进入项目目录启动 Codex CLI
2. 用 `/init` 让项目规则更清晰
3. 把任务目标直接说清楚
4. 让它先读代码、再动手改
5. 改完之后再跑构建或测试
6. 最后让它做一次 review 或总结

这样做的好处是整个流程连在一起，不需要来回切换网页、编辑器和终端。

## 权限配置要怎么选

权限模式没有绝对标准，关键看你当前在做什么。

- 如果只是读代码、看结构，可以更保守一些
- 如果是在自己的项目里高频修改，可以适当放宽
- 如果会涉及联网、安装依赖、执行重要命令，最好保持明确授权

AI 在终端里越能动手，效率越高；但权限越大，也越需要你对结果保持基本检查。

## 为什么它和普通聊天不一样

普通聊天窗口更像“问答工具”，而 Codex CLI 更像“执行型助手”。

差别主要体现在这里：

- 它知道当前工作目录
- 它能直接读项目文件
- 它能运行命令并根据结果继续推进
- 它能把分析、改动和验证串成一个完整流程

这就是为什么很多编码任务在 CLI 里做，会比在单纯聊天窗口里更顺。

## 小结

Codex CLI 的核心价值不只是“能写代码”，而是它把 AI 放进了真正的开发现场。只要你的工作重心在终端和项目目录里，它就不是一个旁观者，而是能持续参与执行的协作者。
